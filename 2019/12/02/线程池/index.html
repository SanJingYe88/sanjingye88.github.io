线程池
Created 2019-12-02| Updated 2019-12-02
Word count: 3.4k|Reading time: 11 min| Post View:

什么是线程池.

  同数据库连接池一样, 预先创建好一定数量的线程, 用来处理任务, 任务处理结束后, 并不销毁线程, 而是放回到线程池中去等待执行下一个任务.

为什么需要线程池 ? ( 使用线程池的好处 )

  ① 降低资源消耗. 重复利用已创建的线程, 降低创建线程和销毁线程所造成的消耗.
  ② 提高任务响应速度. 预先创建好一定数量的线程, 当任务到达时, 不需要等待线程的创建, 就能立即执行.
  ③ 提高线程的可管理性. 使用线程池可以对线程进行统一分配, 调优和监控.

如何预先创建好一堆线程?

  关键点 : 线程的状态. 当线程池的线程刚创建时, 让他们进入阻塞状态, 等待任务的到来. 当任务到来时, 唤醒其中的一个线程, 让它去执行该任务.

  如何让线程进入阻塞状态? 使用阻塞队列 BlockingQueue. 当一个线程调用 BlockingQueue 的 take() 方法获取数据时, 如果这个 Queue 中没有数据, 该线程会被阻塞；同样, 一个线程调用 BlockingQueue 的 put() 方法放数据时, 如果 Queue 满了, 也会被阻塞.

  线程池中每个线程的 run() 方法中, 设置一个循环, 每次都尝试从 BlockingQueue 中获取任务, 如果 Queue 是空的, 就阻塞等待, 如果有任务来了, 就会通知到线程池的某一个线程去处理, 处理完了以后, 再次试图从 BlockingQueue 中获取任务, 就这么一直循环下去.

线程池的类结构

  Java 中的线程池是通过 Executor 框架实现的, 该框架中用到了 Executor, Executors, ExecutorService, ThreadPoolExecutor 这几个类.

—-| Executor 线程池接口
——–| ExecutorService 接口
——–| AbstractExecutorService 类
————-| ThreadPoolExecutor 类 ( 线程池的底层 )
——–| ScheduldExecutorService 类
————-| ScheduledThreadPoolExecutor 类
—-| Executors 线程池工具类

线程池的创建 ( 理论上 )

  使用 Executors线程工具 类中提供的几个静态方法来创建线程池. 其底层也是调用 ThreadPoolExecutor, 只不过参数预先配置好了.

① newSingleThreadExecutor 线程池

  具体实现 : 创建 corePoolSize 和 maximumPoolSize 都为 1 的线程池, 使用 LinkedBlockingQueue；

  特点 :
    创建只有一个线程的线程池.
    如果这个唯一的线程因为异常结束, 那么会有一个新的线程来替代它.
    此线程池可以保证所有任务的执行顺序按照任务的提交顺序执行.

② newFixedThreadPool 线程池

  具体实现 : 创建 corePoolSize 和 maximumPoolSize 都相等 的线程池, 使用 LinkedBlockingQueue；

  特点 :
    创建指定大小的线程池. 线程池的大小一旦达到最大值就会保持不变.
    如果某个线程因为执行异常而结束, 那么线程池会补充一个新线程.

③ newCachedThreadPool 线程池

  具体实现 : 创建 corePoolSize 为 0, maximumPoolSize 为 Integer.MAX_VALUE 的线程池. 使用的 SynchronousQueue 同步队列, 也就是说来了任务就创建线程运行, 当线程空闲超过 60 秒时, 就销毁该线程.

  特点 :
    如果线程池中的线程数量大于了处理任务所需要的线程数量, 那么就会销毁部分空闲的线程.
    如果线程池中的线程数量小于处理任务所需要的线程数量, 那么就会创建一些线程.

线程池的构造方法 / 7 个参数

  在 ThreadPoolExecutor 类中提供了四个构造方法, 不过前三个构造器都是调用的第四个构造器进行的初始化工作.

  创建一个线程池需要的 7 个参数：

① corePoolSize 线程池中的常驻核心线程的数量.
  当任务提交到线程池时, 如果线程池的线程数量还没有达到 corePoolSize, 那么就会新创建的一个线程来执行任务, 如果达到了, 就将任务添加到任务队列中.

② workQueue 任务队列, 用于存放等待执行的任务, 它是一个阻塞队列.

③ maximumPoolSize 线程池允许创建的最大线程数. 如果任务队列满了, 并且已创建的线程数小于最大线程数时, 线程池会再创建新的线程执行任务.
  注意 : 如果使用了无界的任务队列, 这个参数就没什么效果.

④ ThreadFactory 创建线程的工厂. 一般使用默认的即可. 通过线程池工厂, 可以方便的为每一个创建的线程设置具有业务含义的名称, 便于后期排查问题.

⑤ RejectedExecutionHandler 线程饱和时的拒绝策略. 当任务队列满了, 线程池中的线程数量 >= maximumPoolSize 时, 对于新提交的任务采取的处理方式.

⑥ keepAliveTime 多余的空闲线程的存活时间.
  当线程池中的线程数量大于 corePoolSize 时, 那么大于 corePoolSize 这部分的线程, 如果处于空闲状态, 这个时候是不允许它们一直存在的, 它们的最大存活时间就是 keepAliveTime

⑦ TimeUnit keepAliveTime 的单位. 有 7 种取值, 在 TimeUnit 类中有 7 种静态属性： 天, 时, 分, 秒, 毫秒等.

任务队列

  任务队列是 BlockingQueue 接口的实现类. 常见的有 :

  ① ArrayBlockingQueue：基于数组的阻塞队列, 元素按照先进先出（FIFO）的顺序入队, 出队.
    ArrayBlockingQueue 是有界队列. 因为底层实现是数组, 数组在初始化时必须指定大小.

  LinkedBlockingQueue：基于链表的阻塞队列, 元素按照先进先出（FIFO）的顺序入队, 出队.
    如果不指链表的大小, 它默认的大小是 Integer.MAX_VALUE，即2^32 - 1, LinkedBlockingQueue 变为无界队列.
    如果在初始化的时候指定链表大小, 那么它就是有界队列.
    静态工厂方法 Executors.newSingleThreadPool() 和 Executors.newFixedThreadPool() 就是使用了这个队列.

  SynchronousQueue：不存储元素的阻塞队列. 每个插入操作必须得等到另一个线程调用了移除操作后, 该线程才会返回, 否则将一直阻塞.
    静态工厂方法 Executors.newCachedThreadPool 使用了这个队列.

  PriorityBlockingQueue：一个将元素按照优先级排序的阻塞队列, 元素的优先级越高, 将会越先出队列. 这是一个无界队列.

我们可以通过实现该接口自定义任务的拒绝策略.

在 ThreadPoolExecutor 中内置四种处理策略.

① AbortPolicy 通过源码可知, 这是线程池默认的阻塞策略, 处理方式是不执行此任务, 直接抛出一个运行时异常.

② DiscardPolicy 这个方法策略比较有趣, 是个空方法, 处理方式是直接抛弃当前任务, 不执行, 也不报错.

③ DiscardOldestPolicy 从等待队列里面抛弃提交时间最早的一个任务, 然后将新任务加入到队列中.

④ CallerRunsPolicy 如果线程池没有关闭, 则由调用者线程去处理该任务. 该策略既不抛弃任务, 也不会抛出异常.

        比如: 线程 A 提交任务给线程池, 线程池采用此策略, 那么将由线程 A 来执行任务.

线程池的工作原理

  当一个新的任务提交到线程池之后, 线程池的处理流程 :

  ① 提交任务到线程池

  ② 判断核心线程池里的线程是否都在执行任务. ( 即线程池中的线程数目是否大于 corePoolSize )
    如果不是, 则创建一个新的工作线程来执行任务.
    否则执行第三步.

  ③ 判断 BlockingQueue 任务队列是否已经满.
    如果没有满, 则将新提交的任务存储在任务队列中等待执行.
    否则执行第四步.

  ④ 判断线程池的线程是否都处于工作状态. ( 即当前线程池中的线程数目是否大于 maximumPoolSize )
    如果不是, 则创建一个新的工作线程来执行任务.
    否则由任务拒绝策略来处理这个任务.

  ⑤ 拒绝策略会处理该任务. ( 当同时提交的 任务数 > maxPoolSize + workQueue.size() 时, 才会触发拒绝策略 换句话就是 当前线程池中的线程数 > maxPoolSize 时, 才会触发拒绝策略 )

  ⑥ 当一个线程完成任务时, 它会从任务队列中取出下一个任务来执行.

  ⑦ 当一个线程无事可做超过一定的时间 ( keepAliveTime ) 时, 线程池会判断:
    如果当前运行的线程数 > 核心线程数, 那么这个线程会被销毁. 所以线程池的所有任务完成后, 池中的线程数量会自动缩小到核心线程数的大小.

实际工作中如何创建线程池?

  虽然 JUC 提供了好几种线程池, 但是实际工作中只能使用自定义的线程池.

  阿里巴巴开发规范上这么写到 :

  ① [强制] 线程资源必须通过线程池提供, 不允许在应用中自行显示创建线程.

    使用线程池的好处是减少在创建和销毁线程上所消耗的时间以及系统资源的开销, 解决资源不足的问题. 如果不使用线程池, 有可能造成系统创建大量同类线程而导致消耗完内存或者过度切换的问题.

  ② [强制] 线程池不允许使用 Executors 去创建, 而是通过 ThreadPoolExecutor 的方式, 这样的处理方式让写的同学更加明确线程池的运行规则, 规避资源耗尽的风险.

    ① SingleThreadPool 和 FixedThreadPool 使用的任务队列都是 LinkedBlockingQueue, 而它默认的长度为 Integer.MAX_VALUE, 可能会堆积大量的请求, 从而导致 OOM.

    ② CachedThreadPool 和 ScheduledThreadPool 允许创建的线程数量是 Integer.MAX_VALUE, 可能会创建大量的线程, 从而导致 OOM.

任务的提交

  有两种任务提交的方式 : execute 无返回值, submit 有返回值.

  可以使用 execute(Runnable task) 方法向线程池中提交一个任务, 没有返回值. 适用于不需要关注返回值的场景, 只需要将线程丢到线程池中去执行就可以了.

  也可以使用 submit() 方法向线程池中添加任务, 有返回值, 返回值对象是Future.
submit() 方法实际上还是调用的 execute() 方法, 只不过它利用了 Future 来获取任务执行结果.

  submit()方法有三个重载的方法:

     Future submit(Callable task); //传入 Callable 接口的实现类.
     Future submit(Runnable task, T result); //传入 Runable 接口的实现类. 有返回值, 返回值为 T 类型, 通过 Future.get() 可以获取具体的返回值.
    Future<?> submit(Runnable task); //传入 Runable 接口的实现类, 无返回值, 返回值就是 null.
